# Building Product — Compiled Truth

Across a hundred-plus talks, YC's advice on building product converges on one loop: talk to a specific real user, build the smallest ugly thing that could possibly deliver value, ship it fast, and iterate on the same problem/customer rather than chasing a "home run." Speed of iteration beats test coverage, scalability, or polish before product-market fit, and unscalable, manual, handmade work early is not a failure mode but the fastest way to learn what's actually worth building. The same instinct resurfaces once you're building the interface itself: cut to one clear action, speak in the customer's own words, remove friction before asking for commitment, and watch what users actually do rather than trust what they say. The AI-coding era compresses these timelines further — plans in markdown, git discipline, thin-harness/fat-skills, building for the model six months out — but hasn't changed the underlying discipline: founders still need taste, judgment, and direct contact with users to know what's worth building at all.

**State:** #talks: 101 · maturity: strong
**Open threads:** Ship-fast-and-ugly (test coverage doesn't matter pre-PMF, technical debt is fine) vs. fight-for-craft (Stripe's "gravitational pull to mediocrity," MVQP over bare MVP, "no-slop zones") — largely reconciled by stage, but genuinely contested at the margin. Also: minimal AI scaffolding that rides the frontier model up (Anthropic's Claude Code team) vs. heavy custom harnesses/skills investment (Conductor, YC's own G-stack) as the right way to build with agents.
