# How To Design Better AI Apps — Compiled Truth

Most AI features today are "horseless carriages" that bolt AI onto old software UX instead of redesigning the product around what AI actually enables, and Pete Koomen argues the fix is exposing (or auto-personalizing) the system prompt so software can be individually programmed in natural language.

**Key ideas (with timestamps):**
- Gmail's AI drafting agent produces generic, overly formal emails because its hidden system prompt is a one-size-fits-all "lowest common denominator" instruction that every user shares [how-to-design-better-ai-apps @ 05:09](https://youtu.be/WJoZK9sMwvw?t=309)
- Letting a user edit the system prompt themselves (e.g., "you're Pete, a busy 43-year-old father who keeps emails short") produces drafts that actually sound like them [how-to-design-better-ai-apps @ 08:18](https://youtu.be/WJoZK9sMwvw?t=498)
- This is the "AI horseless carriage" problem: like early automobiles that looked like carriages with an engine, most AI features slot a model into an app designed for humans instead of being redesigned around automation [how-to-design-better-ai-apps @ 10:37](https://youtu.be/WJoZK9sMwvw?t=637)
- A demo email-reading agent driven entirely by a plain-English system prompt (label from wife as personal, label boss as priority one, archive sales pitches) shows non-programmers can effectively "program" agents [how-to-design-better-ai-apps @ 13:39](https://youtu.be/WJoZK9sMwvw?t=819)
- Coding agents like Cursor and Windsurf are far ahead of other domain agents because developer tools are "power tools" that grant full, un-nerfed access to the model instead of being hedged against embarrassing the company [how-to-design-better-ai-apps @ 17:59](https://youtu.be/WJoZK9sMwvw?t=1079)
- In the near term most users won't hand-write system prompts, but products can auto-generate and continuously refine a personalized system prompt from past behavior and ongoing feedback, similar to onboarding a new human assistant [how-to-design-better-ai-apps @ 21:41](https://youtu.be/WJoZK9sMwvw?t=1301)
- Agents become genuinely useful for offloading busy work once they have tools (label, archive, draft, and beyond into Slack, calendar, Notion, Jira) rather than being confined to a chatbot Q&A paradigm [how-to-design-better-ai-apps @ 26:05](https://youtu.be/WJoZK9sMwvw?t=1565)
- Founders building AI-native products should ask "how would I design this tool from scratch to offload repetitive work" rather than "how do I insert a chatbot into my existing product" [how-to-design-better-ai-apps @ 29:34](https://youtu.be/WJoZK9sMwvw?t=1774)
